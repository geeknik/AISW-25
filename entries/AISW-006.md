# AISW-006: Hallucinated Security API Usage

| Field | Value |
|-------|-------|
| **Identifier** | AISW-006 |
| **Rank** | #6 of 25 |
| **Severity** | HIGH |

---

## Description

AI models generate code that calls cryptographic, authentication, or sanitization functions with incorrect parameters, deprecated modes, or entirely fabricated API signatures. The code compiles and appears to perform security operations but provides no actual protection.

Examples include using ECB mode for AES, calling a nonexistent `sanitize_html()` function, setting PBKDF2 iterations to training-data defaults from 2012, or using static salts. The failure is silent: the code works, tests pass, and the security gap is only discovered during an active attack.

## Exploit Scenario

A developer requests a password hashing implementation. The assistant generates code using `hashlib.pbkdf2_hmac` but sets iterations to 1,000 (the training-data default from 2012-era examples) instead of the current OWASP-recommended minimum of 600,000. It also uses a static salt derived from the username.

The code passes unit tests because the hashing function works correctly — it just provides negligible brute-force resistance. A database breach six months later results in the full password set being cracked in hours.

## Detection Methods

- Static analysis rules that flag deprecated cryptographic modes, weak iteration counts, and static salts
- Security-focused linting that validates cryptographic API usage against current OWASP recommendations
- Compilation-time checks that verify imported security functions actually exist in the declared library version
- Peer review checklists specifically targeting crypto, auth, and sanitization code generated by AI

## Mitigations

- Provide vetted cryptographic wrapper libraries as the only approved primitives; prohibit direct crypto API usage
- Maintain organization-wide security API usage guides injected as context for all AI coding sessions
- Require security team review for any PR that touches authentication, encryption, or input sanitization logic
- Automated integration tests that validate security properties (e.g., timing resistance, entropy of outputs) not just functional correctness

## Related Mappings

- **CWE-327**: Use of a Broken or Risky Cryptographic Algorithm
- **CWE-916**: Use of Password Hash With Insufficient Computational Effort
- **CWE-330**: Use of Insufficiently Random Values

## Severity Rationale

Ranked HIGH because hallucinated security APIs create a false sense of protection that persists through review. The code looks secure, passes tests, and fails silently. The impact materializes only during an active attack, at which point remediation is too late.
