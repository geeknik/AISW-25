# AISW-001: Authorization Logic Synthesis Failure

| Field | Value |
|-------|-------|
| **Identifier** | AISW-001 |
| **Rank** | #1 of 25 |
| **Severity** | CRITICAL |

---

## Description

AI-generated code implements authentication and happy-path authorization but systematically omits object-level access checks, tenant isolation boundaries, and negative-case deny logic. The model optimizes for functional correctness against sample inputs, producing code that passes integration tests while leaving authorization surfaces open.

This weakness scales because a single flawed pattern in a shared service template propagates to every downstream microservice. The model has no concept of organizational authorization policy — it generates the authorization pattern most common in its training data, which is almost always insufficient for production multi-tenant systems.

## Exploit Scenario

An attacker creates two accounts on a multi-tenant SaaS platform. Service endpoints were generated by a coding assistant that implemented RBAC role checks but not object-level ownership validation. The attacker modifies the resource ID in an API request from their own tenant object to a target tenant's object.

The request passes role-based checks (the attacker holds a valid "user" role) and returns the target's data. Because the same generation pattern was used across 14 resource endpoints, all 14 are vulnerable. The attacker scripts enumeration across all endpoints and exfiltrates the full dataset of another tenant.

## Detection Methods

- Static analysis rules flagging API handlers that query by user-supplied ID without ownership predicates
- Authorization matrix diffing: compare declared RBAC policy against actual code-path enforcement per endpoint
- Negative-case integration tests that attempt cross-tenant and cross-role access on every resource endpoint
- Automated taint analysis from request parameters to database queries checking for unvalidated ID usage

## Mitigations

- Enforce a mandatory authorization middleware layer that cannot be bypassed by individual handlers
- Require generated code to include deny-by-default policies with explicit allow-listing per resource type
- Run automated cross-tenant penetration tests in CI against every API endpoint before merge
- Template all data-access layers with mandatory tenant-scoping predicates injected at the ORM/query-builder level

## Severity Rationale

Ranked #1 because authorization failures are the highest-impact class of web vulnerability, and AI generation makes them multiplicative. A single flawed pattern can instantiate across an entire service mesh in minutes. Detection requires semantic understanding of business logic, which automated tools handle poorly. The blast radius is full data breach per tenant boundary failure.
